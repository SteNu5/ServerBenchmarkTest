Blind SQL Injection

Description
Blind SQL injection is a technique that exploits a vulnerability occurring in the database of an application. This kind of vulnerability is harder to detect than basic SQL injections because no error message will be displayed on the webpage.

Vulnerability found in /action.php

Description HTTP Request cURL command line 
Blind SQL vulnerability via injection in the parameter sql
POST /action.php HTTP/1.1
Host: localhost
Referer: http://localhost/inject.html
Content-Type: application/x-www-form-urlencoded

sql=%27+or+sleep%287%29%3D%27
curl "http://localhost/action.php" -e "http://localhost/inject.html" -d "sql=%27+or+sleep%287%29%3D%27"
Solutions
To protect against SQL injection, user input must not directly be embedded in SQL statements. Instead, user input must be escaped or filtered or parameterized statements must be used.
References
  • OWASP: Blind SQL Injection
  • Wikipedia: SQL injection
  • CWE-89: Improper Neutralization of Special Elements used in an SQL Command ('SQL Injection')

HttpOnly Flag cookie

Description
HttpOnly is an additional flag included in a Set-Cookie HTTP response header. Using the HttpOnly flag when generating a cookie helps mitigate the risk of client side script accessing the protected cookie (if the browser supports it).

Vulnerability found in /

Description HTTP Request cURL command line 
HttpOnly flag is not set in the cookie : wordpress_test_cookie
GET / HTTP/1.1
Host: localhost
curl "http://localhost/"
Solutions
While creation of the cookie, make sure to set the HttpOnly Flag to True.
References
  • OWASP: Testing for Cookies Attributes
  • OWASP: HttpOnly

Secure Flag cookie

Description
The secure flag is an option that can be set by the application server when sending a new cookie to the user within an HTTP Response. The purpose of the secure flag is to prevent cookies from being observed by unauthorized parties due to the transmission of a the cookie in clear text.

Description HTTP Request cURL command line

Solutions
